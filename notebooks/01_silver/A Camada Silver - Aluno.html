🎯 ATO 1 — O Universo de Análise: Selecionando as Empresas

⌛ ~10 min  |  Modo: professor fala

A camada Silver é onde transformamos dado bruto em dado usável. Antes de qualquer transformação, precisamos responder: sobre quais empresas vamos trabalhar?

Não adianta processar todas as ~2.000 empresas registradas na CVM. Queremos apenas as que têm ações negociadas em bolsa, estão em operação e possuem demonstrativos compatíveis com nosso modelo de valuation.

Os Três Filtros Obrigatórios

SIT = 'ATIVO'  &&  TP_MERC = 'BOLSA'  &&  SIT_EMISSOR = 'FASE OPERACIONAL'

SIT = 'ATIVO': empresa com registro ativo na CVM. Descarta extintas, canceladas ou em processo de encerramento.
TP_MERC = 'BOLSA' — o filtro mais crítico: empresas com TP_MERC = NULL têm registro CVM mas não negociam ações. Exemplo: Votorantim Cimentos, registrada por debêntures. Sem cotação, P/L e EV/EBITDA são impossíveis de calcular.
SIT_EMISSOR = 'FASE OPERACIONAL': exclui pré-operacionais e empresas em processo de regularização, cujos demonstrativos são incompletos ou não padronizados.

Setores Excluídos do Universo

📄 ATO 2 — O Problema das Versões: Deduplicando a Bronze

⌛ ~20 min  |  Modo: 10 min conceito + 10 min exercício

A CVM permite que empresas reapresentem seus demonstrativos financeiros. Isso é normal e frequente: corrigem erros tipográficos, ajustam classificações, incorporam notas de auditoria.

Cada reapresentação incrementa o campo "VERSAO". A lógica é sequencial:

MAX("VERSAO"::INTEGER) = 3 → a empresa enviou o documento 3 vezes para aquela data

Por que isso é crítico? Se não deduplicarmos, a mesma conta pode aparecer múltiplas vezes no dataset — com valores diferentes! Uma soma ingnua contaria o mesmo resultado duas ou três vezes, corrompendo qualquer indicador financeiro calculado em seguida.
A solução padrão do pipeline: ROW_NUMBER() OVER(PARTITION BY "CNPJ_CIA", "DT_REFER", "CD_CONTA", "DS_CONTA" ORDER BY "VERSAO"::INTEGER DESC) — ficamos apenas com rn = 1, ou seja, a versão mais recente de cada conta.

🤖 ATO 3 — Python Agnóstico: Uma Query, Várias Tabelas

⌛ ~20 min  |  Modo: acompanhar professor

Uma query “agnóstica” é aquela que não é escrita para uma tabela específica — ela usa um parâmetro no lugar do nome da tabela. O Python injeta o nome correto a cada iteração do loop.

Esse padrão é fundamental em pipelines de dados: evita repetição de código e garante que a mesma lógica se aplique a todos os demonstrativos de forma uniforme. É também a base de como os Colabs da camada Silver funcionam internamente.

🌳 ATO 4 — O Problema da Dupla Contagem e a Função IS_LEAF

⌛ ~20 min  |  Modo: acompanhar professor

A tabela bronze é plana: não há relação hierárquica explícita entre contas pai e filho. Isso significa que 1 (Ativo Total) e 1.01 (Ativo Circulante) existem como linhas independentes — e o valor de 1.01 já está dentro do valor de 1. Somar as duas é dupla contagem.

Vamos demonstrar esse problema com duas queries antes de apresentar a solução do pipeline.

Query 1 — Escolhendo uma Empresa para o Exemplo

Vejamos as empresas disponíveis e selecionemos uma para trabalhar ao vivo:

SELECT DISTINCT "CNPJ_CIA", "DENOM_CIA", "DT_REFER", MAX("VERSAO"::INTEGER) AS versao_maxima FROM layer_01_bronze.dfp_cia_aberta_bpa_con GROUP BY "CNPJ_CIA", "DENOM_CIA", "DT_REFER" ORDER BY "DT_REFER" DESC LIMIT 10;

Anote o CNPJ_CIA e a DT_REFER que iremos usar nas próximas queries.

Query 2 — O Problema: Somar Sem Critério Dá Errado

Com a empresa escolhida, execute a query abaixo. Ela compara dois métodos de totalizar o Ativo e mostra por que um está errado:

WITH dedup AS ( SELECT *, ROW_NUMBER() OVER( PARTITION BY "CNPJ_CIA", "DT_REFER", "CD_CONTA", "DS_CONTA" ORDER BY "VERSAO"::INTEGER DESC ) AS rn FROM layer_01_bronze.dfp_cia_aberta_bpa_con WHERE "CNPJ_CIA" = '<<SEU_CNPJ>>' AND "DT_REFER" = '<<SUA_DATA>>' ), contas AS ( SELECT *, CASE WHEN "ESCALA_MOEDA" = 'MIL' THEN "VL_CONTA"::NUMERIC * 1000 ELSE "VL_CONTA"::NUMERIC END AS vl_normalizado FROM dedup WHERE rn = 1 AND "VL_CONTA"::NUMERIC <> 0 ) -- Método 1: soma ingênua de TODAS as contas (ERRADO — dupla/tripla contagem!) SELECT 'SOMA INGÊNU A (errada)' AS metodo, SUM(vl_normalizado) AS resultado FROM contas UNION ALL -- Método 2: apenas o Ativo Total reportado (CORRETO) SELECT 'ATIVO TOTAL REAL (CD_CONTA = 1)' AS metodo, MAX(vl_normalizado) AS resultado FROM contas WHERE "CD_CONTA" = '1';
💡 O que vocês vão ver: a soma ingênua pode ser 3x ou 4x maior que o Ativo Total real — porque soma 1, 1.01, 1.01.01… tudo junto. A solução é identificar as folhas da árvore (contas sem filhos) e somar só elas.

A Solução: processar_leafs_pandas_rapido() e IS_LEAF

A função usa teoria de conjuntos: pega todos os CD_CONTA reportados pela empresa, gera todos os prefixos ancestrais possíveis e marca como IS_LEAF = True toda conta que não é ancestral de nenhuma outra. Só as folhas devem ser somadas no Power BI ou em qualquer ferramenta de BI.
Contas-pai reconstruídas recebem FLAG_RECONSTRUCAO = True: são geradas sinteticamente pelo pipeline somando os filhos. Isso garante integridade da árvore hierárquica sem apagar ou alterar valores originais das empresas. Safe Mode V11: nunca sobrescrever valor original auditado (abs(valor) > 0.01). Só preenchemos lacunas.
DS_CONTA_REPORTADA preserva o nome original vindo da CVM — linha de auditoria completa. FLAG_NORMALIZACAO = True quando o nome foi padronizado pelo Golden Map.

Colab 3 — Balanço Patrimonial (BP)

📗 Notebook 3
Hierarquia determinística: o nível é determinado pela quantidade de dígitos do CD_CONTA. Nível 1 (1), Nível 2 (1.01), Nível 3 (1.01.01), Nível 4 (1.01.01.01).
Safe Mode V11: pais só são reconstruídos se o valor original for nulo ou zero. Valores reportados com abs(valor) > 0.01 nunca são sobrescritos.
Equação de validação horizontal: Ativo (raiz 1) = Passivo + PL (raiz 2). Diferença > R$ 1.000 é sinalizada como DESBALANCEADO. O threshold de 1.000 é porque a CVM reporta em R$ 1.000 — diferenças menores são arredondamento legítimo.
ST_CONTA_FIXA: preenchido via golden_map_fixa (frequência por CD_CONTA no dataset). Contas reconstruídas recebem 'S' automaticamente — são raízes fixas da taxonomia CVM.

Colab 4 — DRE: A Cascata que Engana

📗 Notebook 4
Raiz '3' não existe na CVM: o menor nível reportado pelas empresas é 3.XX (dois dígitos). O pipeline nunca cria o agregador CD_CONTA = '3'.
A DRE é uma cascata, não uma árvore aditiva pura: cada linha intermediária já embute todas as anteriores. Somar todos os 3.XX gera dupla contagem. A validação correta é sempre par a par: um pai e seus filhos diretos, via JOIN.
EBIT = 3.05: "Resultado Antes do Resultado Financeiro e dos Tributos". Conta fixa (ST_CONTA_FIXA = 'S'), comparável entre todas as empresas.
LPA (Lucro Por Ação): usar 3.99.01.01 (Básico ON) e 3.99.02.01 (Diluído ON). Não usar 3.99 — pai reconstruído que soma básico + diluído = número sem sentido econômico.

Colab 5 — DFC: Equação com Tolerância

📗 Notebook 5
Raiz '6' não existe: a DFC começa em 6.01, 6.02, 6.03, 6.04 (opcional) e 6.05.
Equação fundamental: 6.01 + 6.02 + 6.03 (± 6.04) = 6.05.02 Tolerância de 1% para efeito cambial implícito (CPC 03 §32 — ex: Engie embute o efeito cambial em 6.05 sem reportar 6.04 separadamente). Divergência < 1% com FLAG_RECONSTRUCAO = False → registrar como INFO, não erro.
6.05.02 vs. 1.01.01: sempre usar 6.05.02 para indicadores financeiros — inclui CDBs/LFTs ≤ 90 dias classificados como caixa equivalente pelo CPC 03. 1.01.01 pode ser menor por esse motivo.
CPC 31 / IFRS 5: quando 6.05.02 > 1.01.01 + 1.01.02 com FLAG_RECONSTRUCAO = False, provavelmente há ativos de uma subsidiária em processo de venda classificados em 1.01.08 no BP. Tratar como informativo, não erro do pipeline.
